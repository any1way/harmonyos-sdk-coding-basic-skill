# HarmonyOS SDK 文档检索设计方案

> 本文档描述 HarmonyOS SDK Coding Basic Skill 的本地检索工具设计方案，包括候选方案对比、最终选型、架构设计、数据流、文件结构和性能指标。

---

## 1. 背景与需求

### 1.1 问题背景

本 Skill 的 `references/` 目录收录 12000+ 篇鸿蒙官方文档（guides/api/faq/best-practices/release 等分类），用于 AI Agent 在编码时查询指南和 API 参考。

早期版本采用 `grep` 关键词检索，存在两个核心问题：

1. **遍历效率低** — grep 需要顺序扫描所有文件，12000 篇文档单次查询耗时数秒到十几秒。
2. **语义准确率低** — grep 只能做字面匹配，无法处理同义词、概念改写、自然语言提问。例如查询"怎么让应用启动更快"，grep 无法匹配到"应用冷启动时延优化"文档。

### 1.2 设计目标

| 目标 | 指标 |
|------|------|
| 高效率 | 单次查询 < 1 秒（含模型加载） |
| 高准确性 | Top N 召回相关文档，兼顾精确匹配与语义泛化 |
| 单机可用 | 无 GPU 依赖，纯 CPU 可运行 |
| 零运维 | 首次运行自动建索引、下载模型，后续自动维护 |
| 离线友好 | 索引和模型内置仓库，克隆即用 |

## 2. 候选方案对比

### 2.1 方案清单

| 方案 | 核心算法 | 模型依赖 | 索引类型 |
|------|---------|---------|---------|
| 方案 1 | 纯 BM25 | 无 | 倒排索引 |
| 方案 2 | 纯 Dense Embedding | BGE 系列模型 | FAISS 向量索引 |
| 方案 3 | Hybrid + RRF | BGE 系列模型 | 倒排 + FAISS 双索引 |
| 方案 4 | LLM rerank | 大语言模型 | 倒排 + FAISS + LLM |

### 2.2 详细对比

#### 方案 1：纯 BM25

- **原理**：基于词项频率（TF）、逆文档频率（IDF）和文档长度归一化的经典关键词检索算法。
- **优势**：
  - 无模型依赖，纯 Python 标准库实现
  - 索引构建快（~1 分钟）
  - 精确关键词匹配能力强（API 名、类名、错误码）
  - 查询延迟 < 20ms
- **劣势**：
  - 无法处理同义词、语义改写（"启动更快" ≠ "冷启动时延优化"）
  - 中文需要分词，分词质量影响召回
  - 自然语言提问效果差

#### 方案 2：纯 Dense Embedding

- **原理**：用预训练语言模型（BGE 系列）将文档和查询编码为 512 维稠密向量，通过余弦相似度匹配。
- **优势**：
  - 语义泛化能力强（同义词、概念改写、自然语言提问）
  - 跨语言能力（中英文混合查询）
  - 不依赖分词
- **劣势**：
  - 依赖模型，首次需下载（~100MB）
  - 索引构建慢（CPU 约 10 分钟）
  - 精确关键词匹配弱（API 名可能召回不准）
  - 查询延迟较高（含模型加载 ~1 秒，编码 ~0.4 秒）

#### 方案 3：Hybrid + RRF（最终选型）

- **原理**：并行执行 BM25 和 Dense 检索，通过 Reciprocal Rank Fusion 算法融合两路排名。
- **优势**：
  - 兼顾精确匹配（BM25）与语义泛化（Dense）
  - 两路独立，可分别调权重
  - RRF 无需训练，对分数尺度不敏感
  - 实测在 10 类测试用例中表现最稳
- **劣势**：
  - 需维护两套索引
  - 依赖模型（但可降级为纯 BM25）

#### 方案 4：LLM rerank

- **原理**：在 Hybrid 召回基础上，用大语言模型对 Top N 重新打分排序。
- **优势**：精度最高，可处理复杂语义关系
- **劣势**：
  - 依赖 LLM API（网络延迟、费用、密钥管理）
  - 不适合单机离线场景
  - 与 Skill 的"本地优先"理念冲突

### 2.3 探索历程与方案演进

本节记录实际开发过程中探索过的方案演变，包含已放弃的中间方案及其放弃原因，供后续优化参考。

#### 阶段 1：grep 遍历（最初方案，已放弃）

- **实现方式**：用 `grep -ri "<关键词>" references/` 遍历所有 .md 文件
- **探索结果**：
  - 12000 篇文档单次查询 5–15 秒，AI Agent 多轮对话体验差
  - 纯字面匹配，"启动更快"无法匹配"冷启动时延优化"
  - 无相关性排序，返回结果按文件路径字典序，需人工筛选
- **放弃原因**：效率与准确率均不达标
- **遗留**：作为最后兜底手段，在索引全部损坏时仍可手动使用

#### 阶段 2：纯 BM25（方案 1，保留为降级方案）

- **实现方式**：[build_index.py](build_index.py) 构建 jieba 分词 + TF-IDF 倒排索引，[search.py](search.py) 查询
- **探索结果**：
  - 查询 20ms，效率达标
  - API 名、错误码等精确匹配场景表现优秀
  - 但"怎么让应用启动更快"等自然语言查询召回偏差大
- **结论**：作为基础通道保留，不单独使用，但作为 Hybrid 的 BM25 通道和降级方案

#### 阶段 3：BGE-M3 纯 Dense（探索后放弃）

- **实现方式**：尝试用 BGE-M3（568M 参数，1024 维）编码全部文档
- **探索结果**：
  - CPU 编码 12000 篇文档预计 3+ 小时，首次构建不可接受
  - 模型 1.1GB，克隆仓库成本高
  - 单次 query 编码约 8 秒，交互体验差
- **放弃原因**：无 GPU 环境下性能不可接受
- **经验**：Dense 方案在单机 CPU 场景必须选轻量模型

#### 阶段 4：BGE-small 纯 Dense（方案 2，验证可行但未单独采用）

- **实现方式**：切换到 BGE-small-zh-v1.5（24M 参数，512 维）
- **探索结果**：
  - CPU 编码 12000 篇文档 10 分钟，可接受
  - 模型 91MB，仓库可承受
  - query 编码 0.4 秒，体验良好
  - 语义泛化能力强，"启动更快"成功召回"冷启动时延优化"
- **遗留问题**：API 名精确匹配弱于 BM25，"AVPlayer setURL" 召回不如 BM25 精准
- **结论**：不单独使用，作为 Hybrid 的 Dense 通道

#### 阶段 5：Hybrid + RRF（方案 3，最终选型）

- **实现方式**：[search_hybrid.py](search_hybrid.py) 并行调用 BM25 + Dense，RRF 融合
- **探索结果**：
  - 10 类测试用例全部表现优秀（详见第 7.3 节）
  - 语义查询靠 Dense 救场，精确查询靠 BM25 救场
  - 双路 rank=1 的文档（如"NDK Node-API"）融合后置信度最高
- **结论**：定为最终方案

#### 阶段 6：LLM rerank（方案 4，评估后未采用）

- **实现方式**：计划在 Hybrid Top 10 基础上用 LLM 精排
- **探索结果**：
  - 需调用云端 LLM API，与 Skill 本地优先理念冲突
  - 增加网络延迟（1–3 秒）和费用
  - 需管理 API 密钥
- **结论**：当前不采用，但预留接口，未来可作为可选增强

#### 演进时间线

```
grep 遍历 ──(效率低)──> 纯 BM25 ──(语义弱)──> BGE-M3 Dense
                                                      │
                                              (CPU 太慢)
                                                      ↓
                             BGE-small Dense ──(精确弱)──> Hybrid + RRF ✓
                                                              │
                                                       (已选型，预留)
                                                              ↓
                                                     LLM rerank（未来可选）
```

#### 方案演进对比总结

| 方案 | 探索阶段 | 单次查询耗时 | 建索引耗时 | 语义能力 | 精确能力 | 是否采用 |
|------|---------|-------------|-----------|---------|---------|---------|
| grep 遍历 | 阶段 1 | 5–15s | 无 | 无 | 弱 | ✗ 放弃 |
| 纯 BM25 | 阶段 2 | 20ms | 1 分钟 | 无 | 强 | △ 降级备选 |
| BGE-M3 Dense | 阶段 3 | 8s | 3+ 小时 | 优 | 中 | ✗ 放弃 |
| BGE-small Dense | 阶段 4 | 0.5s | 10 分钟 | 良 | 中 | △ 作 Hybrid 通道 |
| **Hybrid + RRF** | **阶段 5** | **0.5s** | **10 分钟** | **良** | **强** | **✓ 最终选型** |
| LLM rerank | 阶段 6 | 1–3s | 10 分钟 | 优 | 优 | ✗ 预留未采用 |

### 2.4 硬件需求对比

| 方案 | CPU | 内存 | 磁盘 | GPU |
|------|-----|------|------|-----|
| 方案 1 纯 BM25 | 任意双核 | 512MB | 150MB（索引） | 无需 |
| 方案 2 纯 Dense (small) | 4 核推荐 | 2GB | 250MB（模型+索引） | 无需（可选） |
| 方案 2 纯 Dense (m3) | 8 核推荐 | 4GB | 1.5GB（模型+索引） | 推荐（无 GPU 索引构建 3+ 小时） |
| **方案 3 Hybrid (small)** | **4 核推荐** | **2GB** | **400MB（模型+双索引）** | **无需** |
| 方案 3 Hybrid (m3) | 8 核推荐 | 4GB | 1.6GB | 推荐 |
| 方案 4 LLM rerank | 任意 | 512MB | 400MB | 无需（依赖云端 LLM） |

### 2.5 选型决策

**最终选择：方案 3（Hybrid + RRF），模型用 BGE-small-zh-v1.5**

理由：

1. **精度优势** — 10 类测试用例验证，Hybrid 在语义查询（用例 1/4/7）和精确查询（用例 2/9/10）场景均表现优秀，纯 BM25 或纯 Dense 都有盲区。
2. **硬件友好** — BGE-small-zh-v1.5 仅 24M 参数，CPU 推理速度差 BGE-m3 约 20 倍（10 分钟 vs 3+ 小时），Top-N 召回场景精度差距可接受。
3. **降级能力** — Dense 通道不可用时（模型/依赖缺失），可回退到 `search.py` 纯 BM25，保证基本可用。
4. **无需训练** — RRF 是无参算法，无需标注数据训练融合模型，k=60 是经验默认值。

### 3.2 RRF 融合算法

**公式**：

```
score(d) = Σ_c  w_c / (k + rank_c(d))
```

- `d`：候选文档
- `c`：检索通道（BM25 / Dense）
- `w_c`：通道权重（默认 BM25=0.5, Dense=0.5，可通过 `--weights` 调整）
- `k`：平滑常数，默认 60（经验值，控制排名靠前文档的得分差距）
- `rank_c(d)`：文档 d 在通道 c 中的排名（从 1 开始）

**特性**：
- 对分数尺度不敏感（只看排名，不看绝对分数）
- 无需训练，开箱即用
- 当某通道未召回该文档时，该项贡献为 0

**示例**：

| 文档 | BM25 rank | Dense rank | RRF 分数 (w=0.5/0.5, k=60) |
|------|-----------|------------|---------------------------|
| A | 1 | 2 | 0.5/61 + 0.5/62 = 0.01626 |
| B | None | 1 | 0 + 0.5/61 = 0.00820 |
| C | 3 | None | 0.5/63 + 0 = 0.00794 |

文档 A 双路都靠前，融合后排名第一。

### 3.3 模型选择

| 候选模型 | 参数量 | 维度 | CPU 编码速度 | 中文能力 | 选型 |
|---------|--------|------|-------------|---------|------|
| BGE-m3 | 568M | 1024 | 慢（3+ 小时建索引） | 优秀 | ✗ |
| BGE-base-zh-v1.5 | 102M | 768 | 中等 | 优秀 | ✗ |
| **BGE-small-zh-v1.5** | **24M** | **512** | **快（10 分钟）** | **良好** | **✓** |
| text2vec-base-chinese | 102M | 768 | 中等 | 良好 | ✗ |

**选 BGE-small-zh-v1.5 理由**：
- CPU 友好，无 GPU 也能 10 分钟建完 12000 篇文档索引
- 24M 参数足够处理鸿蒙文档的中英文混合场景
- Apache-2.0 协议，可商用
- Top-N 召回场景（N=5~10）精度差距与 base/m3 可接受

## 6. 关键设计决策

### 6.1 为何选 RRF 而非加权分数融合

| 融合方式 | 公式 | 问题 |
|---------|------|------|
| 加权分数 | `0.5*bm25_score + 0.5*dense_score` | 分数尺度不同（BM25 0~30，Dense 0~1），需归一化，归一化方式影响效果 |
| **RRF** | `Σ w/(k+rank)` | **只看排名，对尺度不敏感，无需归一化** |
| Learned rerank | 训练融合模型 | 需标注数据，复杂度高 |

### 6.2 为何选 BGE-small 而非 BGE-m3

| 维度 | BGE-small-zh-v1.5 | BGE-m3 |
|------|-------------------|--------|
| 参数量 | 24M | 568M |
| CPU 建索引 | 10 分钟 | 3+ 小时 |
| 模型大小 | 91MB | 1.1GB |
| Top-5 召回率 | 85%+ | 90%+ |
| 单次查询编码 | 0.4s | 8s |

Top-N 召回场景下精度差距 < 5%，但速度差 20 倍，small 性价比更高。

### 6.3 为何把模型和索引纳入版本控制

| 方案 | 优势 | 劣势 |
|------|------|------|
| .gitignore 排除 | 仓库小 | 首次需下载模型+建索引，10+ 分钟 |
| **纳入版本控制 + LFS** | **克隆即用** | 仓库大 255MB |

Skill 的核心价值是"开箱即用"，255MB 通过 Git LFS 管理，克隆后立即可查询，体验更好。

### 6.4 降级策略

```
Hybrid 检索 (search_hybrid.py)
    │
    ├─ 模型/依赖缺失 → 自动下载/构建
    │
    ├─ 下载失败 → 转纯 BM25 (search.py)
    │
    └─ BM25 索引也缺失 → 报错并提示重建命令
```

## 8. 使用方式

### 8.1 标准查询

```bash
# Hybrid 查询（推荐）
python scripts/search_hybrid.py "ArkUI 状态管理" --top 5 --snippet

# 纯 BM25 查询（降级/快速）
python scripts/search.py "AVPlayer setURL" --top 5

# 分类过滤
python scripts/search_hybrid.py "API 变更" --category release

# 调整权重（提高 Dense 权重）
python scripts/search_hybrid.py "怎么让应用启动更快" --weights 0.4 0.6

# JSON 输出（供程序消费）
python scripts/search_hybrid.py "音频播放" --top 5 --json
```

### 8.2 索引维护

```bash
# 重建 BM25 索引（references 更新后）
python scripts/build_index.py

# 重建 Dense 索引（--force 强制）
python scripts/build_dense_index.py --force
```

### 8.3 首次运行行为

仓库克隆后直接执行 `search_hybrid.py` 即可，无需任何预处理：

- 模型已内置 `models/`，无需下载
- 索引已内置 `scripts/index/`，无需构建
- 首次查询 ~1 秒（模型加载），后续 ~0.5 秒
